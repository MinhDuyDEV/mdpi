# Ideation Frameworks Reference

Use these frameworks selectively. Pick the lens that fits the idea — don't mechanically run every framework.

## SCAMPER

- **Substitute:** What component, material, or process could you swap out?
- **Combine:** What two things that don't usually go together would create something new?
- **Adapt:** What ideas from other industries, domains, or time periods could you borrow?
- **Modify (Magnify/Minimize):** What if you made it 10x bigger? 10x smaller?
- **Put to other uses:** Who else could use this? What other problems could it solve?
- **Eliminate:** What happens if you remove a feature entirely? What's the zero-configuration version?
- **Reverse/Rearrange:** What if you did the steps in opposite order? What if the user did the work instead of the system?

**Best for:** Improving or reimagining existing products/features.

## How Might We (HMW)

Reframe problems as opportunities: "How might we [desired outcome] for [specific user] without [key constraint]?"

**Good HMW:** Narrow enough to be actionable, broad enough to allow creative solutions, contains a tension or constraint.

**Bad HMW:** Too broad ("make users happy"), too narrow ("add a button"), solution-embedded ("build a chatbot").

**Best for:** Reframing stuck thinking.

## First Principles Thinking

1. **What do we know is true?** (not assumed, actually true)
2. **What are we assuming?** List every assumption.
3. **Which assumptions can we challenge?**
4. **Rebuild from the truths.**

**Best for:** Breaking out of incremental thinking.

## Jobs to Be Done (JTBD)

Focus on what the user is trying to accomplish:
- **Functional job:** What task?
- **Emotional job:** How do they want to feel?
- **Social job:** How do they want to be perceived?

Format: "When I [situation], I want to [motivation], so I can [expected outcome]."

**Best for:** Understanding the real problem.

## Constraint-Based Ideation

Deliberately impose constraints: time, feature count, tech restriction, cost, audience, scale.

**Best for:** Cutting through complexity.

## Pre-mortem

Imagine the idea has already failed. List every plausible reason. Which are preventable? Which would kill the project?

**Best for:** Phase 2 evaluation. Stress-testing ideas that feel good but haven't been pressure-tested.

## Analogous Inspiration

Look at how other domains solved similar problems. Find *structural* similarities, not surface-level ones.

**Best for:** Phase 1 expansion. Generating genuinely different variations.
